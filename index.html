07 - Controle avançado de alterações git checkout nomedoarquivo: Volta o arquivo para o seu esado anterior git reset HEAD nomedoarquivo: Volta o arquivo ao estado unstaged, o que equivale a um arquivo antes de ser adicionado (add) para comiit git reset identificadorDoCommit: Volta o status do git para o commit do identificador selecionado. Podemos capturar um identificador através do comando git log git revert identificadorDoCommit: Faz um novo commit que desfaz o commit correspondente ao identificador. git stash: guarda as modificações que não foram comitadas. Desta forma podemos utilizar outros comandos de modificação de commits e depois abrir o stash para prosseguir com o que havíamos guardado. git stash list: lista o que tem guardado em seu stash git stash pop: retoma os últimos dados salvos no stash. git stash apply identificadorDoStash: retoma os dados guardados no stash puxando pelo identificador. git stash drop: apaga as alterações guardadas no stash. git bisect start: Começa um processo de triagem de commits para achar um commit específico. Após startar o bisect deve-se inserir os comandos "git bisect bad identificadorDoCommit" e "git bisect good identificadorDoCommit" (lembrando que podemos inserir "git bisect HEAD" caso queiramos usar o último commit como base) sendo que o "good" é a versão que queremos e "bad" é a versão que não queremos. Após isso o bisect irá retornar versões anteriores do arquivo, nós devemos analisar estas versões dentro do arquivo e dizer se esta é a versão boa ou a ruim digitando "git bisect good" ou "git bisect bad" de acordo com o resultado. Ao final o bisect irá finalizar com o commit que queremos e irá retornar seu identificador. 08 - Contribuição com opensource: Técnicas avançadas e produtividade com o GIT Fork: Em muitos projetos não será possível ter permissão para dar push diretamente no repositório, porém nada impede que façamos uma versão nossa do projeto paralelamente ao original. Para isso utilizamos o recurso "Fork" do próprio github. Esse recurso cria uma cópia do projeto no nosso repositório reomoto onde teremos controle total. Neste instante podemos criar o clone em nossa máquina a partir deste repositório sem problemas, adicionando melhorias ao projeto conforme formos desenvolvendo. Porém podemos sugerir ao dono original do projeto que adicione as nossas alterações caso ele se interesse, a esse processo damos o nome de "Pull Request". O pull request pode ser feito a partir da própria interface gráfica do github, mas falaremos melhor dele mais adiante. Contudo o projeto original receberá outras melhorias do autor e de outros colaboradores, para ficar a par destas melhorias é recomendado que façamos uma cópia do repositório original para continuar recebendo commites e pushes, essa cópia relaciona o repositório original ao fork e deve ser feita com o seguinte comando: git remote add original [caminho_do_repositorio_remoto] Sempre que quisermos carregar as alterações do repositório original utilizamos o comando git fetch original e logo em seguida o comando git merge original/master para dar merge nas alterações. Pull request: O pull request é feito através da interface gráfica do próprio github (ou outros programas de gerenciamento de GIT). Quando fazemos os pull request deixamos uma mensagem dizendo o que foi modificado e aguardamos a requisição ser aceita ou não. Sempre que um pull request é criado, uma URL é atribuída a ele. Dessa maneira, é possível visualizar através do navegador as informações do pull request criado. Essa URL possui o seguinte formato: https://github.com/[dono_do_projeto]/[projeto]/pull/[id_do_pull_request] Entrando nesse endereço, é possível visualizar também as modificações introduzidas pelo pull request, que é como é possível avaliar se o pull request enviado está OK ou não. O Git nos permite conferir o diff de diversas maneiras, a mais comum e prática é a dada no próprio pull request, podemos fazer essa modificação manualmente, é prático porém mais trabalhoso. Outra forma de fazer é diretamente pelo GIT, sem precisar copiar código. Uma vez que o pull request foi gerado, basta baixá-lo para o nosso projeto e realizarmos o git merge para trazer as alterações para uma branch nossa. Com isso, bastaria primeiro fazer: git remote add [alias_do_repositorio_remoto] [url_do_repositorio_remoto] E em seguida fazermos o git fetch. Com isso, já teremos todas as alterações baixadas em nosso computador. A seguir, basta realizar o git merge [alias_do_repositorio]/[branch]. Isso trará para a branch atual os commits existentes no repositório remoto. Por fim, temos que colocar os commits em nosso repositório remoto. Para isso, basta executar o git push e, com isso, introduzimos os commits do pull request de outro desenvolvedor ao nosso projeto.
